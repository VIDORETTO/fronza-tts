# Planejamento Completo — App Python TTS Multi-API com Cota Grátis, Controle de Limite e Fallback Automático

## 1. Objetivo do projeto

Criar um aplicativo em Python para gerar áudio a partir de texto usando APIs de Text-to-Speech, priorizando provedores que oferecem algum tipo de uso gratuito via API.

O app deve permitir que o usuário:

* Digite ou cole um texto.
* Escolha o idioma.
* Escolha o provedor de voz ou deixe no modo automático.
* Escolha modelo, voz e configurações de áudio quando disponíveis.
* Gere áudio via API.
* Reproduza o áudio dentro do app.
* Baixe o arquivo gerado.
* Veja histórico de gerações.
* Veja quanto já usou de cada provedor.
* Veja se a cota é mensal, diária, inicial, manual ou incerta.
* Evite gastar dinheiro acidentalmente.
* Faça fallback automático para outro provedor quando a cota grátis estiver acabando, quando houver rate limit ou quando uma API falhar.

O diferencial do app não será apenas gerar áudio. O diferencial será ter um **gerenciador inteligente de cotas grátis** e um **roteador de fallback** que escolhe o melhor provedor disponível no momento.

---

## 2. Ideia central

O app deve funcionar como uma camada intermediária entre o usuário e várias APIs de TTS.

Fluxo geral:

```text
Usuário digita texto
        ↓
App calcula tamanho, idioma, estimativa de uso e risco de cobrança
        ↓
Quota Manager verifica saldo/cota de cada provedor
        ↓
Router escolhe o melhor provedor disponível
        ↓
Provider Adapter chama a API correta
        ↓
Áudio é salvo localmente
        ↓
Uso é registrado no banco
        ↓
Painel de limite é atualizado
        ↓
Se falhar, o app tenta o próximo provedor
```

A estrutura deve ser pensada para que novos provedores possam ser adicionados facilmente depois, sem reescrever o app inteiro.

---

## 3. Ponto mais importante: nem toda cota grátis reseta mensalmente

O app precisa diferenciar tipos de limite.

Não pode existir apenas uma coluna chamada “limite mensal”, porque isso ficaria errado para vários provedores.

A estrutura correta é ter uma política de reset por provedor.

## 3.1 Tipos de limite

| Tipo de limite          | Como funciona                                                         | Exemplo de uso no app                                        |
| ----------------------- | --------------------------------------------------------------------- | ------------------------------------------------------------ |
| `monthly_billing_cycle` | Reseta todo mês ou no ciclo de cobrança                               | ElevenLabs, Cartesia, Inworld em planos com créditos mensais |
| `daily_rate_limit`      | Reseta por dia, normalmente por projeto ou modelo                     | Gemini API                                                   |
| `per_minute_rate_limit` | Limite por minuto ou janela curta de tempo                            | Gemini, Cartesia, Smallest e outros                          |
| `one_time_trial_credit` | Crédito grátis inicial, não necessariamente volta todo mês            | Smallest, Async ou outros provedores com crédito de teste    |
| `manual_balance`        | O app não consegue consultar automaticamente; usuário informa o saldo | Smallest, Async, provedores com painel sem endpoint de saldo |
| `unknown`               | Não há informação confiável suficiente                                | Provedor fica com uso restrito                               |
| `paid_topup`            | Créditos comprados separadamente                                      | Não usar automaticamente no modo grátis                      |
| `pay_as_you_go`         | Cobra conforme uso                                                    | Bloquear no modo “não gastar”                                |

---

## 3.2 Regra de segurança

O app deve sempre partir da regra:

> Se o reset da cota não for confirmado, o app não deve assumir que é mensal.

Isso evita que o sistema use uma API achando que o saldo voltou, quando na verdade o crédito grátis era apenas inicial.

## 6. Filosofia do app

O app deve ter 3 modos de operação:

## 6.1 Modo seguro grátis

Nome interno:

```yaml
free_only_mode: true
```

Esse é o modo padrão.

Nesse modo:

* Nunca usar provedor sem API key.
* Nunca usar provedor com cobrança pay-as-you-go sem saldo grátis confirmado.
* Nunca usar top-up pago automaticamente.
* Nunca assumir reset mensal se não houver confirmação.
* Nunca gerar texto grande em provedor com cota incerta.
* Nunca continuar tentando infinitamente após erro.
* Nunca usar uma API que retornou erro de cobrança, quota exceeded ou billing required.
* Sempre avisar o usuário antes de usar provedor com cota incerta.

Esse modo deve ser o padrão porque o objetivo do app é aproveitar cotas grátis sem gastar por acidente.

---

## 6.2 Modo automático com fallback

Nesse modo o usuário só digita o texto, escolhe idioma e clica em gerar.

O sistema decide:

* Qual provedor suporta o idioma.
* Qual provedor tem saldo/cota disponível.
* Qual provedor tem menor risco de cobrança.
* Qual provedor está menos próximo do limite.
* Qual provedor falhou recentemente.
* Qual provedor deve ser pulado.

---

## 6.3 Modo manual

Nesse modo o usuário força um provedor específico.

Mesmo assim o app deve avisar:

* Se a cota estiver baixa.
* Se a cota for incerta.
* Se o provedor for pay-as-you-go.
* Se o reset não for mensal.
* Se existe risco de cobrança.

O app pode permitir gerar mesmo assim, mas deve exigir confirmação explícita quando houver risco.

## 7.2 Por que Streamlit?

Streamlit é o melhor caminho para um MVP rápido porque permite criar facilmente:

* Campo de texto grande.
* Contador de caracteres.
* Dropdown de idioma.
* Dropdown de provedor.
* Dropdown de voz.
* Botão de gerar.
* Player de áudio.
* Tabela de limites.
* Histórico.
* Painel de configurações.

Depois, se quiser transformar em app desktop mais bonito, pode migrar a UI para:

* Flet
* PySide6
* Tauri com backend Python
* FastAPI + frontend web

Mas para começar, Streamlit é mais rápido e suficiente.

## 8.2 Regra de ouro

A interface nunca deve chamar APIs diretamente.

Errado:

```python
# Errado
if provider == "elevenlabs":
    requests.post(...)
```

Certo:

```python
# Certo
result = tts_service.generate(request)
```

A UI só fala com o `TTSService`.

O `TTSService` fala com o `TTSRouter`.

O `TTSRouter` escolhe o provedor.

O provedor real fica escondido dentro de um adapter.

## 23.1 Políticas

### `monthly_billing_cycle`

Usado quando a cota é mensal.

Regras:

* Se a API informar data de renovação, usar a data oficial.
* Se não informar, usar data manual configurada.
* Se não houver data, mostrar “ciclo mensal não confirmado”.

### `daily_rate_limit`

Usado quando o limite é diário.

Regras:

* Reset diário.
* Não somar isso como “crédito mensal”.
* Mostrar no painel como limite de requests/tokens por dia.

### `per_minute_rate_limit`

Usado para limites de RPM/TPM.

Regras:

* Controlar janela curta local.
* Se receber 429, esperar ou fazer fallback.
* Não consumir cota mensal.

### `one_time_trial_credit`

Usado para crédito grátis inicial.

Regras:

* Nunca resetar automaticamente.
* Só atualizar se o usuário informar novo saldo.
* Se acabar, marcar como esgotado.

### `manual_balance`

Usado quando o app não sabe consultar saldo.

Regras:

* Usuário informa saldo.
* App desconta estimativa.
* App exibe confiança média/baixa.
* App pede revisão periódica.

### `paid_topup`

Usado quando o usuário compra créditos.

Regras:

* Não usar no modo grátis.
* Só usar se o usuário ativar `allow_paid_balance`.

### `pay_as_you_go`

Usado quando não existe cota grátis clara.

Regras:

* Bloquear no modo grátis.
* Exigir confirmação para qualquer geração.

## 24.1 Motivos para fallback

* Erro 5xx.
* Voz indisponível.
* Idioma indisponível.
* Chave inválida.
* Billing required.
* Modelo fora do ar.

## 24.2 Ordem inicial recomendada

```text

3. Smallest.ai

5. Inworld
```

## 24.3 Ranking automático

O app pode calcular uma pontuação:

```text
score =
  prioridade_configurada
  + bonus_cota_oficial
  + bonus_cota_restante
  + bonus_reset_mensal_confirmado
  + bonus_suporte_idioma
  + bonus_qualidade_usuario
```

## 25. Fluxo completo de geração

## 25.1 Antes da geração

O app deve:

1. Receber texto.
2. Remover espaços desnecessários.
3. Contar caracteres.
4. Estimar tokens.
5. Estimar duração.
6. Detectar idioma, se necessário.
7. Verificar modo escolhido.
8. Verificar provedores habilitados.
9. Remover provedores sem API key.
10. Remover provedores sem suporte ao idioma.
11. Consultar ou carregar cota.
12. Estimar consumo em cada provedor.
13. Aplicar reset policy.
14. Aplicar regra de risco.
15. Escolher provedor.
16. Avisar se houver risco.

---

## 25.2 Durante a geração

O app deve:

1. Chamar provider adapter.
2. Aplicar timeout.
3. Aplicar retry controlado.
4. Se erro recuperável, tentar fallback.
5. Se erro de autenticação, bloquear provedor.
6. Se erro de cobrança, bloquear provedor.
7. Se erro de rate limit, marcar cooldown.
8. Se erro de voz, tentar voz padrão.
9. Se erro de idioma, tentar próximo provedor.

---

## 25.3 Depois da geração

O app deve:

1. Salvar áudio.
2. Salvar metadados.
3. Registrar uso no ledger.
4. Atualizar cota.
5. Registrar fallback se ocorreu.
6. Atualizar histórico.
7. Mostrar resultado.
8. Permitir download.
9. Permitir regenerar com outro provedor.

## 34. Prompt final para a IA implementar

Use este prompt para uma IA de código:

```markdown
Você deve implementar um app Python chamado TTS Fallback App.

Objetivo:
Criar um app local em Python para gerar áudio TTS usando múltiplas APIs com cota grátis ou crédito inicial. O app deve permitir inserir texto, selecionar idioma, provedor, modelo e voz, gerar áudio, reproduzir/baixar o resultado, acompanhar limites grátis e fazer fallback automático para outro provedor quando a cota estiver perto de acabar ou quando ocorrer erro de quota, rate limit, timeout, billing required ou indisponibilidade.

Stack:
- Python 3.11+
- Streamlit
- SQLite
- SQLModel ou SQLAlchemy
- httpx
- pydantic
- python-dotenv
- PyYAML
- tenacity
- loguru
- pydub
- ffmpeg

Arquitetura obrigatória:
- UI não pode chamar APIs diretamente.
- Criar TTSService.
- Criar TTSRouter.
- Criar Provider Adapters.
- Criar QuotaManager.
- Criar ResetPolicyEngine.
- Criar FallbackEngine.
- Criar UsageLedger.
- Criar AudioStorage.
- Criar TextChunker.

Provedores do MVP:
1. ElevenLabs
2. Cartesia
3. Smallest.ai
4. Gemini API
5. Inworld
6. Async Voice API

Regra crítica:
Nem toda cota grátis reseta mensalmente.
O app deve diferenciar:
- monthly_billing_cycle
- daily_rate_limit
- per_minute_rate_limit
- one_time_trial_credit
- manual_balance
- paid_topup
- pay_as_you_go
- unknown

Modo padrão:
free_only_mode = true

Nesse modo:
- Nunca gastar dinheiro automaticamente.
- Nunca usar pay-as-you-go sem confirmação.
- Nunca usar top-up pago automaticamente.
- Nunca assumir reset mensal sem confirmação.
- Nunca gerar texto grande com cota incerta.
- Bloquear provedor com billing_required.
- Fazer fallback em quota_exceeded, rate_limited, timeout, server_error, unsupported_language, unsupported_voice e billing_required.

Telas obrigatórias:
1. Gerar áudio
2. Limites e cotas
3. Provedores
4. Histórico
5. Comparação de qualidade

Banco:
Criar tabelas:
- providers
- api_keys
- quota_snapshots
- manual_balances
- usage_ledger
- generations
- fallback_events

Critério de pronto:
- Consigo digitar texto em português.
- Consigo gerar áudio.
- O app escolhe automaticamente um provedor seguro.
- O app mostra se a cota é mensal, diária, manual, inicial ou incerta.
- Se a cota está acabando, o app pula para outro provedor.
- Se uma API retorna 429, quota exceeded ou billing required, o app faz fallback.
- O áudio é salvo localmente.
- O histórico mostra provedor, modelo, voz e consumo.
- O painel de limites mostra usado, restante, reset e fonte da informação.
- Nenhuma API key aparece em logs ou tela completa.
```

## 35. Veredito técnico

A melhor estrutura é:

```text
Streamlit UI
+ TTSService
+ TTSRouter
+ Provider Adapters
+ QuotaManager
+ ResetPolicyEngine
+ UsageLedger SQLite
+ FallbackEngine
+ AudioStorage
```

O ponto mais importante é o app ter uma visão clara de:

```text
Cota disponível
+ tipo de reset
+ fonte da informação
+ confiança da informação
+ risco de cobrança
```

Sem isso, o app pode gerar áudio, mas não será seguro.

Com isso, o app vira um gerenciador inteligente de APIs TTS grátis, capaz de usar cada provedor da forma correta, sem confundir crédito mensal com crédito inicial e sem gastar dinheiro sem perceber.
